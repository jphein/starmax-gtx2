Concept

How a face works

A custom face is a static image container — a background plus glyph/asset files — with widgets positioned on top. Each widget is bound to one data field the watch already tracks. There is no free-form data channel: to show an external value you smuggle it into a field the firmware renders.

Build the container.

A background image + per-digit glyph fonts, packed into the native dial format (RGB565 + LZ4). Each image asset's LZ4 match length is capped at 512 bytes or the watch's minimal decoder garbles it.

Install once, over BLE.

Streamed to the watch via the D-plane push (chunked). Installs are reliable on a contention-free radio; the node serialises pushes so concurrent watches don't collide.

Update live — no re-push.

Values ride the RTC. set_time_custom(hour, minute, second, day) writes the clock fields; the widgets bound to them re-render on the watch. Changing the value is one small frame, never a new image.

Components

The six widget types

Every element on a face is one of these. A widget's kind is how it draws; its binding is what data drives it.

text

A number drawn from a per-digit glyph font, bound to a field (hour, day, heart…). The workhorse. Each font renders at its own baked size — reuse a big font and you get big digits.

clock · kW number · BPM

arc

A curved progress bar / ring that fills proportional to its bound field. Used for the battery ring — and it works on a pushed field too (see below).

battery ring · gauges

pointer

A rotating hand sprite — analog clock hands, driven by hour/min/second.

analog hands

array

A repeated set of elements — e.g. a step-history mini bar chart (steplist).

step history

curved

Text laid out along a curved baseline, for text that follows the round bezel.

curved labels

icon

A static image — the full-screen background, or a small baked graphic like the heart glyph next to a BPM number.

background · icons

Decision guide

Which field for what

The one rule that governs everything: a pushed value only stays if it lands on a field that holds — one the firmware doesn't tick or reformat. That's why a clock uses seconds happily but a data value never can.

A number that must hold

Use day (1–31). It only changes at midnight, so a pushed value stays. This is where integer kW lives. It's a 2-digit field, so it alone covers 1–31.

day · proven · live now

The clock

Use hour + min (+ second if you want ticking seconds). Seconds are fine here because they tick — that's what a clock does.

hour · min · second

Extra digits

Only day (1–31) and month (1–12) carry pushed values — both proven clean on-glass. week/weekday looked promising but the firmware derives it from the date and ignores pushes, so it's out. Two carriers is enough for a one-decimal readout.

day + month · proven 2026-07-16

A gauge / bar

An arc widget. Bind it to battery for the charge ring, or to day for a kW gauge that fills with the pushed value.

arc → battery · arc → day

Live watch stats

heart, step, distance, calorie, battery — the watch fills these from its own sensors. Display them; you can't push them.

sensor · display-only

Never for data

second (ticks away within a second) and date (formats as "MM-DD", e.g. "0731"). Both look tempting; both destroy a pushed number.

second · date · avoid
Pushing values

Getting a number onto the watch

There's no weather or arbitrary-text binding, so an external value has to ride a clock/calendar field. The live grid-watts face keeps the real clock on hour/min and puts kilowatts on day.

Grid-watts encoding (live today)

hour, min, second = real time (clock stays correct) · day = round(kW), an integer 1–31 · every watt change just re-writes the RTC via set_time_custom, no image push. Caveat: nothing else may sync real time to that watch or it overwrites day with the real date — the scheduler excludes the live-kW watch so its driver is the sole clock writer.

The decimal: day.month = "1.1 kW"

One decimal is enough, so the readout is day (integer kW) + a baked "." + month (tenths) — shipped & live on the daily 2026-07-16. Both hold. The driver pushes month = tenths directly (0–9): RTC month is nominally 1–12, but the display widget renders the raw byte, so month = 0 shows "0" (HW-confirmed) — so an exact "3.0 kW" is reachable with no offset font or trick. week/weekday can't help (the firmware derives it from the date). The fuller total-watts idea is shelved — one decimal covers the grid range cleanly. Staleness ("how old is this reading?") also can't live on the watch — no spare holdable field — so it's an HA-dashboard tile instead.

Your question

Can a bar show a pushed variable?

Yes. The arc widget renders a progress bar that fills proportional to its bound field — and the binding can be a pushed field, not just a sensor.

Battery ring vs. pushed gauge — same widget, different source

The green battery ring is an arc bound to battery (a sensor). Bind an arc to day instead and it becomes a kW gauge — it fills to round(kW) / 31, driven by the same push that sets the number. Toggle kW bar in the preview to see it: a red arc that tracks the pushed watts alongside the number.

Because day tops out at 31, a raw day-bound arc maxes there; scale the visual fill in the renderer to your real ceiling (e.g. 12 kW) for a full sweep.
